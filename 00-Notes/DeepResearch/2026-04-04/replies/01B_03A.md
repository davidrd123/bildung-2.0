## Q_set01 Prompt B — Frame Audit: Reserved Namespace or Real Layer?

Evidence base used here only: `patterns/handle-schema.md`, `00-Notes/process.md`, `texts/zeitmauer/threads/goethe-leibniz-frame.md`, and `00-Notes/2026-04-04-earned-distillation-from-peirce-thinking-methods-chat.md`.

### Classification

Reserved namespace.

### Why this classification

`patterns/handle-schema.md` is explicit that `frame:*` is reserved and that the frame layer is not yet a fully earned canonical object layer. It repeats the point later even more bluntly: the namespace is stable, but the object layer remains experimental. On the canonical schema alone, that already puts the answer on the reserved-namespace side.

`00-Notes/process.md` confirms the same status at the workflow level. In the architectural-thread table, the frame layer is described as reserved in schema and experimental in practice, not yet earned as a crystallized object layer. So the repo’s own process document does not treat frames as a settled layer with standing authority.

The strongest counter-pressure in the allowed evidence set is `texts/zeitmauer/threads/goethe-leibniz-frame.md`. But that file matters precisely because it is a `thread:*` dossier, not a `frame:*` object. It shows that frame-like reading is happening in practice, yet it is being housed inside the thread layer. The repo has experimental frame-work; it does not yet have a first-class frame layer.

The Peirce distillation note sharpens the brake. It says actual `frame:*` use should stay experimental until repeated frame-reads survive reuse and reveal their blind spots, and it warns against letting an imported lens become retroactive justification for architecture not yet earned by practice. That is an explicit anti-promotion rule, not a quiet admission that the layer already exists.

### Rival reading

A fair rival reading is that the repo already has the functional beginnings of a frame layer. The schema defines `frame:*`, gives example handles, and even supplies promotion gates. The Goethe/Leibniz dossier behaves like a frame document in all but name. On that reading, the layer exists practically, even if the filenames and canonical layer lag behind.

### Weakening evidence

This verdict would weaken if the canonical files stopped describing frames as reserved or experimental, or if dossiers like Goethe/Leibniz were promoted into first-class `frame:*` documents rather than being carried by `thread:*` files.

Reserved namespace is therefore the stronger classification. What exists now is live frame-pressure under explicit experimental discipline, not a real layer.

## Q_set03 Prompt A — Handle Schema Audit: Earned Grammar or Anticipatory Ontology?

Evidence base used here only: `patterns/handle-schema.md`, `00-Notes/process.md`, `texts/zeitmauer/source/handles.yaml`, `texts/zeitmauer/threads/README.md`, and `00-Notes/genealogy-to-instrument.md`.

### Classification

Earned grammar.

### Why this classification

`patterns/handle-schema.md` is not just aspirational prose. It defines concrete handle types, relation types, maturity statuses, and promotion gates. It also locates current reality at least at a “Convention only” phase, which already assumes active use of handles in notes, YAML, and journals. That is grammar: a declared syntax for stable reference, relation, and maturity.

`00-Notes/process.md` shows that this grammar has entered method, not just design talk. It gives `patterns/handle-schema.md` high authority in the repo’s precedence order, and it operationalizes the relation vocabulary directly. The clearest example is the instruction that `challenges` is an active relation type and that evidence handles can carry negative evidence. That is handle grammar doing real argumentative work.

`texts/zeitmauer/source/handles.yaml` is the decisive proof that the grammar is earned. It densely instantiates section handles, chunk handles, note handles, thread handles, evidence handles, and explicit relations across the Zeitmauer pilot. This is not a toy schema with one or two examples. It is a live index with repeated object forms and repeated relational use.

`texts/zeitmauer/threads/README.md` reinforces that these are not merely naming conveniences. It says thread dossiers are the “first-class commentary layer” for the subproject. So at least one major object type in the schema has clearly crossed from grammar into working layer.

At the same time, `00-Notes/genealogy-to-instrument.md` keeps the classification from drifting into triumphalism. It says the repo already has the materials of the instrument, but “not yet the full architecture,” and then lists what is still missing: a first-class frame layer, a cross-project atomic layer, stable relation vocabulary as shared practice, residue as first-class data, operational memory, a dynamic view layer, and system-level maturity criteria. That means the schema is earned as grammar, but not yet complete as ontology.

### Rival reading

A fair rival reading is anticipatory ontology. `patterns/handle-schema.md` reaches beyond current practice into future phases such as dynamic view and agent integration, and `genealogy-to-instrument.md` explicitly says several major layers are still missing. Read that way, the schema names more of the intended instrument than the repo has fully built.

### Weakening evidence

The earned-grammar verdict would weaken if the schema remained abstract. It holds because the grammar is canonized in `patterns/handle-schema.md`, folded into workflow in `process.md`, instantiated densely in `handles.yaml`, and backed by a subproject README that recognizes thread dossiers as a real commentary layer.

So the stronger answer is earned grammar, with anticipatory ontology at the edges. The handle system is real enough to organize work now, but not yet complete enough to count as a fully realized repo-wide ontology.
