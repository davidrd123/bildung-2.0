## Classified Claims

* **1. A structured session ledger is ready to test now — `test now`**
  This is the strongest near-term claim. The repo already has a real but inconsistent session-offload layer: one session ends with **“The Mental State”** and **“Resume Points,”** another with **“Mental Model at Session End”** and **“What’s Next,”** another with **“What to carry forward”** and **“Memory notes saved.”** The Mythos chat’s ledger proposal looks less like a new architecture than a normalization of an existing need. That need is explicit elsewhere too: `genealogy-to-instrument.md` says the repo is still weak on operational agent memory and that journals are not yet traversable enough for re-entry; `process.md` sets the practical standard as discoverability and asks whether a cold start can reconstruct why a decision was made. The weaker, earned formulation is: **a structured ledger is ready for a small living-layer pilot, not for repo-wide standardization or promotion into `patterns/`.** ([GitHub][2])

* **2. `re_entry_instructions` is the key ledger field and directly addresses re-entry — `test now`**
  This claim survives cleanly. The Mythos chat explicitly names `re_entry_instructions` as the key innovation: not a summary, but a briefing written by the outgoing collaborator for the incoming one. That maps tightly onto an already live repo pressure. The Warburg chat frames faster re-entry after a gap as one of the actual criteria in the journal-vs-handle experiment, and later says the project already experiences nonlinear gains when a session begins by reading the journal, glossary, and cross-domain threads. `AGENTS.md` says the shortest path back into useful work is the latest batch file, local journal, local glossary, and active thread dossiers. `re_entry_instructions` is the field that can compress exactly that path for a given edge. The weaker formulation is: **this is likely the highest-value field in the ledger pilot, not yet a proven universal key for every subproject.** ([GitHub][3])

* **3. `formation_notes` should be captured in the moment as raw traces and reviewed later like `seed` material — `test now`**
  This survives, but only in its weakened form. The chat already does the weakening: it argues that the high-resolution pivot evaporates if it is not captured immediately, but that the capture should be treated as raw trace, not as a finding. That fits the repo’s own maturity logic unusually well. `handle-schema.md` already treats `seed`, `tentative`, and `open` as legitimate provisional states, and the Peirce distillation shows the repo knows how to keep productive material below the promotion line until more evidence arrives. The Cassirer journal also names a close-reading capture gap: paragraph-scale gains were disappearing into conversation before finding a stable home. So the claim is testable now **if** it stays optional, raw, and reviewable. The weaker formulation is: **allow one brief raw formation trace inside the ledger pilot, reviewed later like seed material; do not treat it as authoritative self-knowledge about the collaborator.** ([GitHub][3])

* **4. Journals should allow a lightweight `formation` heading when a batch genuinely changed the reader’s grip on the material — `hold in living layer`**
  This is the first claim that feels premature at journal level. There is real support for it: the charter says the project aims at practices that form the reader, and that journals preserve how understanding develops. The Mythos chat’s proposed `formation` heading is lightweight and optional. But the repo evidence still cuts against promoting this into normal journal practice now. The Cassirer journal explicitly separated `journal.md` from broader extrapolative note types so it could stay source-bound and workflow-bound. `right-now.md` warns against turning hardening moves into universal overhead, and `CLAUDE.md` warns about template grooves: repeated headings can start generating notes shaped by the form rather than the material. The weaker formulation is: **do not add a standing `formation` heading to canonical journals yet; keep formation traces in session ledgers or occasional ad hoc review-note language when directly tied to a source resistance and clearly useful on reread.** ([GitHub][3])

## Best Near-Term Tests

* **Run a 2-session ledger pilot, not a repo-wide change.**
  Use the next 2 sessions in two different trace environments — ideally one journal-heavy (`erkenntnisproblem`) and one handle-rich (`zeitmauer`). Keep the pilot in the living/session layer, since the review surface explicitly says not to update `patterns/`, `process.md`, `right-now.md`, or canonical files from this pass alone. Minimal fields: `state_on_entry`, `what_shifted`, `unresolved`, `re_entry_instructions`, and optional `formation_notes`. Then start the following session from the ledger plus the normal AGENTS path (latest batch, local journal, glossary, thread dossier) and see whether re-derivation drops and the live edge is recovered faster. ([GitHub][4])

* **Treat `re_entry_instructions` as the decisive subtest.**
  The incoming collaborator should try to resume from that field first, not from the full chat export. The practical question is simple: does it get the session back to the real edge faster than a generic summary does? This is the most direct way to test claim 2. ([GitHub][3])

* **Test `formation_notes` only as one-line raw pivots.**
  Limit them to one brief sentence in the form: “I came in trying to close X to Y; the text resisted because Z.” On review 1–2 sessions later, decide whether the trace preserved something that would otherwise have vanished. If it does not improve reread or later filtering, drop it. ([GitHub][3])

## Claims Most Likely To Cause Overbuild

* **Claim 4 is the main overbuild risk.**
  A standing `formation` heading in journals is exactly the kind of reusable format that can groove too fast, especially in journals already carrying dense source-bound and review-bound work. ([GitHub][5])

* **Claim 1 becomes risky only if it jumps from pilot to architecture too quickly.**
  The ledger looks promising because it regularizes an existing practice. It becomes unearned if it turns into a new object type or standing repo-wide requirement before the 2-session pilot shows real re-entry gains. ([GitHub][4])

* **Claim 3 becomes risky if it stops being raw trace and starts pretending to be collaborator psychology.**
  It works only while it stays seed-like, provisional, and source-resistance-based. Once it becomes rhetoric about “formation environments” or authoritative statements about a collaborator’s inner state, it outruns repo evidence. ([GitHub][3])

The cleanest outcome from this pass is: **claims 1 and 2 are ready for a small living-layer pilot now; claim 3 is also testable, but only as optional seed-status raw trace inside that pilot; claim 4 should stay out of canonical journal practice for now.**
